# LocalCut Pro 产品说明书与项目审核报告

版本：V1.1  
日期：2026-06-04  
依据：基于当前代码库的功能、实现与构建审核

## 1. 产品概述

LocalCut Pro 是一款运行在浏览器端的本地图片素材切割工作台。用户可以导入一张或多张图片作为图层，在画布中移动、缩放和切换图层，并创建多个裁剪区域。裁剪区域支持矩形、圆形、三角形、菱形、星形、心形、自定义多边形和魔棒智能选区。用户还可以使用画笔、橡皮和文字标注进行轻量编辑，最终将单个区域导出为图片，或将多个区域批量打包为 ZIP。

产品核心价值是：在不上传图片、不依赖后端的前提下，完成多图层合成、局部选区、轻量标注和批量素材导出。

## 2. 产品定位

当前产品定位为“本地多图层素材切割工具”，介于截图裁剪工具和专业图片编辑器之间。

它适合做：

- 从一张或多张素材图中批量提取局部图片。
- 对局部区域做快速标注、涂抹、擦除后导出。
- 通过魔棒或多边形处理不规则选区。
- 在多图层合成画面中按统一区域导出结果。

它暂时不适合承诺为：

- 完整 Photoshop 类图像编辑器。
- 可长期保存和恢复的项目制设计工具。
- 精准抠图或复杂图像分割工具。
- 面向移动端触控优先的图片编辑产品。

## 3. 目标用户

- 电商与运营人员：从商品图、活动图、组合图中快速切出多个交付素材。
- 设计与内容制作人员：对图片局部区域进行快速圈选、文字说明和导出。
- 游戏与教学素材整理人员：从大图、截图或素材集里提取多个规则或不规则区域。
- 本地隐私敏感用户：希望图片只在浏览器端处理，不经过服务器。

## 4. 核心使用流程

1. 用户上传或拖拽导入图片。
2. 系统将图片作为图层加入画布，新增图层会自动偏移并成为当前活动图层。
3. 用户可选择、移动、缩放图层，也可隐藏或删除图层。
4. 用户使用形状、多边形或魔棒创建裁剪区域。
5. 用户可移动、缩放、重命名、复制、粘贴、删除裁剪区域。
6. 用户可添加文字、使用画笔或橡皮修改活动图层。
7. 用户设置导出格式、质量、DPR 和单区域输出尺寸。
8. 用户导出当前区域，或批量导出全部/勾选区域为 ZIP。

## 6. 功能说明

### 6.1 图片与图层

用户可以多次上传或拖拽图片。每张图片会创建一个图层，包含：

- 图层名称。
- 原始图片对象。
- 可被画笔/橡皮修改的工作画布。
- 画布位置 x/y。
- 缩放比例 scaleX/scaleY。
- 可见性 visible。

图层支持：

- 点击右侧图层列表切换活动图层。
- 点击可见性按钮隐藏或显示图层。
- 删除图层。
- 在画布中拖拽移动图层。
- 拖拽图层控制点缩放图层。
- Ctrl + 双击画布中的图层，切换活动图层。

### 6.2 裁剪区域

支持创建以下区域：

- 矩形。
- 圆形。
- 三角形。
- 菱形。
- 星形。
- 心形。
- 自定义多边形。
- 魔棒生成的多边形。

区域支持：

- 画布点击选中。
- 右侧列表选中。
- 拖拽移动。
- 控制点缩放。
- 右侧输入精确编辑 x/y/宽/高。
- 重命名。
- 删除。
- 复制和粘贴。
- 方向键 1px 微移。
- Shift + 方向键 10px 微移。

自定义多边形支持：

- 连续点击添加顶点。
- 点击起点、双击或 Enter 完成。
- Escape 取消。
- 拖拽顶点调整。
- 双击边线新增顶点。
- 右键顶点删除。

### 6.3 魔棒智能选区

魔棒基于活动图层的工作画布执行颜色连通区域识别：

- 以点击像素为种子点。
- 根据容差计算颜色距离。
- 使用连通填充生成 mask。
- 追踪区域轮廓。
- 简化轮廓点。
- 生成可编辑的自定义多边形区域。

当前魔棒适合背景清晰、颜色边界明显的素材。复杂纹理、渐变或半透明边界可能需要手工修点。

### 6.4 画笔与橡皮

画笔和橡皮作用于当前活动图层的工作画布。

画笔支持：

- 大小：1-100px。
- 颜色：任意颜色。

橡皮支持：

- 大小：1-150px。

画笔和橡皮效果会参与导出。开启“原图”后，画布预览和导出会优先使用原始图像而不是工作画布。

### 6.5 文字标注

文字标注是全局对象，不绑定具体图层。文字支持：

- 点击画布创建。
- 选中、移动、缩放文字框。
- 编辑内容。
- 修改字号。
- 设置常规/粗体。
- 修改颜色。

导出区域时，若文字框与区域重叠，文字会被裁剪到区域形状内。

### 6.6 导出

支持格式：

- PNG。
- JPEG。
- WebP。

JPEG 和 WebP 支持质量设置。PNG 不使用质量参数。

单区域导出：

- 导出当前选中区域。
- 可使用自定义输出宽高。
- 可通过 DPR 提高导出像素密度。
- 文件名使用区域名称。

批量导出：

- 若未勾选区域，则导出全部区域。
- 若已勾选区域，则导出勾选区域。
- 输出 ZIP，默认名称为 `output.zip`。
- 当前批量导出仍使用区域原始宽高，不应用单区域自定义输出尺寸。

当前导出逻辑会合成所有可见图层，再按区域形状裁剪，并绘制区域内文字。

## 7. 快捷键

- Ctrl + Z：撤销。
- Ctrl + Y：重做。
- Ctrl + C：复制选中区域。
- Ctrl + V：粘贴区域。
- Delete / Backspace：删除选中区域。
- 方向键：微移选中区域 1px。
- Shift + 方向键：微移选中区域 10px。
- Ctrl + 双击：切换到点击位置下的图层。
- Enter：完成自定义多边形。
- Escape：取消自定义多边形。
- Space + 拖拽：平移画布。
- 鼠标滚轮：缩放画布。

## 8. 数据与隐私

当前项目没有后端服务、登录账号、远程存储或接口请求。图片读取、图层编辑、选区生成、文字绘制和导出都在浏览器端完成。

需要明确的是，当前项目没有“项目文件保存/打开”功能。刷新页面或关闭页面后，图片、图层、区域、文字和历史记录都会丢失。

## 9. 当前实现边界

- 产品主要面向桌面浏览器，未做移动端触控适配。
- 图层、区域、文字都位于同一画布坐标系，但区域和文字没有绑定到具体图层。
- 图层缩放已经存在于画布交互中，但部分导出和编辑逻辑仍需要进一步校准。
- 历史记录已经扩展到图层、文字和工作画布 ImageData，但会带来明显内存成本。
- 魔棒依赖同步 `getImageData` 和主线程计算，大图上可能卡顿。
- 批量导出不支持统一尺寸策略。
- 没有导出命名规则配置。
- 没有图层重命名、排序、锁定、透明度、混合模式。

## 10. 项目审核结论

### 10.1 总体结论

项目已经从单图裁剪 MVP 升级为多图层素材切割工具，功能方向是成立的，构建也能通过。新增的方向键微移、多图层、图层可见性、图层移动/缩放和更完整的历史记录，确实回应了上一版最值得补强的点。

但当前实现处在“功能先接上，模型还没完全收拢”的阶段。主要问题不是安全漏洞，而是多图层之后的坐标、导出、历史记录和 UI 行为一致性还没有完全闭环。如果继续堆功能，后面会越来越难维护。

### 10.2 已验证事项

- `npm run build` 通过。
- 项目仍为纯前端实现，没有发现网络请求、后端密钥或明显数据外传逻辑。
- 依赖较少，主要是 Vue、Pinia、JSZip、file-saver，供应链面相对简单。
- 产品主链路仍清晰：导入、编辑、选区、导出。

### 10.3 主要问题与风险

#### P1：图层缩放后导出坐标不正确

位置：`src/composables/useExport.ts:23`

导出合成图层时只使用 `region.x - layer.x` 和 `region.width/height` 作为源图裁剪参数，没有除以 `scaleX/scaleY`，也没有按缩放后的图层尺寸计算重叠范围。图层被缩放后，导出的区域会和画布预览不一致。

建议：统一定义“世界坐标 -> 图层源图坐标”的转换函数，导出、魔棒、画笔、命中测试都复用同一套转换逻辑。

#### P1：魔棒没有处理图层偏移与缩放

位置：`src/composables/useCanvasEngine.ts:503`

魔棒点击点来自画布世界坐标，但直接拿这个坐标去活动图层 `workingCanvas` 取像素。当前只在图层位于原点、未缩放时相对正确。图层移动或缩放后，魔棒选区会偏移或失败。

建议：点击点应先转换为活动图层本地坐标，再执行 `getImageData` 和选区计算；生成的轮廓点再从图层本地坐标转换回世界坐标。

#### P1：画笔与橡皮没有处理图层缩放

位置：`src/composables/useCanvasEngine.ts:391`

画笔只做了 `ix - layer.x`、`iy - layer.y`，没有除以图层缩放比例。图层缩放后，涂抹位置、笔刷大小和视觉预览会不一致。

建议：写入工作画布时使用 `(worldX - layer.x) / scaleX`、`(worldY - layer.y) / scaleY`，笔刷大小也需要根据缩放策略明确是屏幕尺寸、世界尺寸还是源图像素尺寸。

#### P1：删除区域快照顺序错误，清空区域不可撤销

位置：`src/components/RightSidebar.vue:291`、`src/components/RightSidebar.vue:304`

删除按钮先执行 `editor.deleteRegion(r.id)`，再执行 `history.snapshot()`，这会把删除后的状态压入撤销栈，导致撤销行为不符合预期。清空按钮直接 `splice(0)`，没有快照，无法撤销。

建议：所有破坏性操作都先 `history.snapshot()`，再修改状态；清空、删除图层、删除文字、切换可见性也应统一走 action，避免模板内直接改 store。

#### P2：历史记录保存完整 ImageData，内存风险很高

位置：`src/stores/history.ts:42`、`src/stores/history.ts:73`

每次快照都会对每个图层执行 `getImageData`，最多保留 50 次。大图、多图层、连续画笔操作会迅速占用大量内存，并造成主线程卡顿。

建议：短期降低历史栈上限，并对画笔拖拽只在 mouseup 后快照一次；中期引入操作级 history 或图层脏块快照；长期考虑 command pattern。

#### P2：新增图层后视图不会自动适配新图层

位置：`src/App.vue:15`、`src/components/CanvasWorkspace.vue:173`、`src/composables/useCanvasEngine.ts:145`

上传图片现在调用 `editor.addLayer(img)`，但没有调用 `engine.fitToCanvas()`。`loadImage()` 仍保留旧单图逻辑，却不再被主流程使用。结果是新增首图或新活动图层后，画布视图可能不自动聚焦到该图层。

建议：将“添加图层后适配视图”的逻辑收口到 CanvasWorkspace 或 editor action，避免保留未使用的旧入口。

#### P2：批量导出按钮文案在未勾选时显示 0 项

位置：`src/components/RightSidebar.vue:157`、`src/components/RightSidebar.vue:308`

逻辑上未勾选时会导出全部区域，但按钮文案显示 `批量导出 — 0 项`。这会造成用户误判。

建议：`checkedCount` 为 0 时显示全部区域数量，或文案改为“批量导出全部”。

#### P2：图层模型可能已经超过当前产品的自然复杂度

位置：`src/stores/editor.ts:8`、`src/composables/useCanvasEngine.ts:636`

图层支持位置、缩放、可见性、删除、切换，但还缺少排序、锁定、重命名、透明度、图层绑定关系和导出坐标闭环。现在它已经显著增加复杂度，却没有完整形成用户心智。

建议：先把图层限定为“多图片合成底图”能力，完成坐标一致性和导出一致性后，再考虑图层排序、锁定等扩展。

#### P3：模板中直接操作 store，状态入口分散

位置：`src/components/RightSidebar.vue:200`、`src/components/RightSidebar.vue:291`、`src/components/RightSidebar.vue:304`

模板直接执行 `layer.visible = !layer.visible`、`editor.regions.splice(0)`、`editor.deleteRegion()`，导致快照、校验、渲染调度和副作用很容易漏掉。

建议：把这些操作收敛为 store action 或 composable 方法，例如 `toggleLayerVisible`、`clearRegions`、`deleteRegionWithHistory`。

#### P3：UI 图标和设计体系混杂

位置：`src/components/LeftSidebar.vue`、`src/components/TopBar.vue`、`src/components/RightSidebar.vue`

当前同时使用字符符号、emoji、文本按钮。作为工具型产品可以先接受，但若要走专业工具方向，应统一图标体系、按钮尺寸、tooltip 和可访问性。

建议：短期保留现状；中期引入统一图标组件，避免 emoji 在不同系统渲染不一致。

### 10.4 漏洞与安全评估

当前没有发现高危安全漏洞。主要原因：

- 没有后端接口和鉴权逻辑。
- 没有远程图片请求流程。
- 没有 `v-html` 或明显 XSS 注入点。
- 用户文字通过 Canvas 绘制，不作为 HTML 注入。
- 文件读取仅通过本地 FileReader。

需要注意的安全/稳定性边界：

- 超大图片可能导致浏览器内存暴涨。
- 历史记录保存完整 ImageData，会扩大内存风险。
- 导出的 ZIP 文件名使用区域名称，虽然 JSZip 一般能处理字符串，但仍建议过滤 `/ \ : * ? " < > |` 等特殊字符，避免跨平台解压体验异常。
- 没有文件尺寸和像素尺寸提示，用户导入超大图时缺少预警。

## 11. 非功能性要求

### 11.1 性能

- 常见 2K-4K 图片应保持基本流畅。
- 超大图片、多个图层、连续画笔和魔棒操作需要性能提示。
- 历史记录应避免每次操作完整复制所有图层像素。

### 11.2 兼容性

- 面向现代桌面浏览器。
- 依赖 Canvas、FileReader、Blob、ResizeObserver、download 等能力。
- 暂不承诺移动端触摸体验。

### 11.3 可用性

- 上传图片后应自动聚焦可编辑区域。
- 图层缩放、移动、导出必须保证预览与结果一致。
- 清空、删除、隐藏等破坏性操作应支持撤销。
- 批量导出数量文案应与实际导出逻辑一致。

## 12. 验收标准

- 上传 PNG/JPEG/WebP 后生成图层并正常显示。
- 多次上传可形成多图层，并可切换、隐藏、删除、移动、缩放图层。
- 图层移动或缩放后，画布预览、魔棒、画笔、橡皮和导出结果坐标一致。
- 可创建并编辑所有预设形状区域。
- 可创建、完成、取消和编辑自定义多边形区域。
- 可通过魔棒生成可编辑区域。
- 可添加、移动、缩放和编辑文字。
- 可单独导出区域为 PNG/JPEG/WebP。
- 可批量导出全部或勾选区域为 ZIP。
- 撤销/重做覆盖区域、文字、图层可见性、图层位置/缩放、画笔/橡皮关键操作。
- 清空、删除区域、删除图层等破坏性操作可撤销。
- `npm run build` 通过。

## 13. 后续优化优先级

### 13.1 必须优先修复

- 修复图层缩放下的导出坐标。
- 修复图层移动/缩放下的魔棒坐标。
- 修复图层缩放下的画笔/橡皮坐标。
- 修复删除区域和清空区域的撤销问题。
- 修复批量导出按钮数量文案。

### 13.2 短期优化

- 新增图片后自动 fit 到画布。
- 限制或提示超大图片。
- 降低历史记录内存占用。
- 区域名称导出前做文件名清洗。
- 将模板内直接状态修改收口为 action。

### 13.3 中期优化

## 14. 技术实现概览

- `src/App.vue`：应用布局、上传入口、全局快捷键和方向键微移。
- `src/components/TopBar.vue`：品牌、设置、快捷键说明。
- `src/components/LeftSidebar.vue`：上传、工具选择、文字列表。
- `src/components/CanvasWorkspace.vue`：画布容器、标尺、拖拽上传、原图开关、画布引擎接入。
- `src/components/RightSidebar.vue`：图层、属性、导出、区域列表。
- `src/composables/useCanvasEngine.ts`：画布渲染、命中测试、区域/文字/图层交互、画笔、魔棒、缩放和平移。
- `src/composables/useExport.ts`：可见图层合成、区域裁剪、文字绘制、图片和 ZIP 导出。
- `src/composables/magicWandUtils.ts`：颜色连通区域选择、轮廓追踪、轮廓简化。
- `src/composables/shapeUtils.ts`：形状路径绘制与区域命名。
- `src/stores/editor.ts`：图层、区域、文字、工具和画布设置状态。
- `src/stores/history.ts`：撤销/重做快照。
- `src/stores/export.ts`：导出设置。

## 15. 最终建议

当前阶段不建议继续大规模增加新功能。更稳的路线是先做“多图层一致性收敛”：统一坐标转换、统一状态 action、统一历史记录策略。等图层移动/缩放后的预览、编辑、魔棒和导出都一致后，再推进项目保存、批量命名、图层排序等增强能力。
